# F4 · 多尺寸屏幕适配状态

> 状态快照：2026-07-17  
> 当前发布承诺：**只支持小皮的 iPad Pro 11 英寸（A2759）横屏、添加到主屏幕后的 PWA。**  
> 13 英寸、其他 iPad、竖屏、Split View、iPhone 和桌面浏览器均不属于当前发布验收范围。

这份文件记录“已经具备的通用能力”和“真正完成支持的设备”之间的差别。后续恢复多尺寸适配时，以本文为入口，不把早期外壳截图误当成完整产品已经适配。

## 2. 已经完成、未来可直接复用的基础

- 核心舞台永久使用 1194×834 逻辑坐标，页面不会按设备宽度重新排列内部元素。
- `scale = min(W / 1194, H / 834)`、居中偏移和逻辑坐标反算已有纯函数与单元测试。
- `ResizeObserver` 会随可用区域变化重新计算舞台；角色拖拽和持久化保存逻辑坐标，而不是设备像素。
- `.f4-safe-area` 已读取 `env(safe-area-inset-*)`；外围 `.f4-bleed` 负责填充不同宽高比的额外区域。
- 竖屏引导和低于舒适缩放阈值的全屏提示已经实现。
- 阶段 B 留有 1194×834、1366×1024、竖屏和 Split View 的历史截图，可用于恢复工作时对照缩放外壳。
- 阶段 G 已完成相对 scope/start URL、横屏 standalone manifest、锁定母图派生图标和全部 F4 PNG 的 Workbox 预缓存；这些是跨尺寸可复用的 PWA 基础，不代表新增设备支持。

## 3. 当前不能宣称完成的部分

- 现有 13 英寸、竖屏和 Split View 截图主要来自阶段 B 的开发外壳；它们证明缩放策略可行，不证明后来增加的真实首页和未来学习页面都已适配。
- 1366×1024 的浏览器模拟不等于 iPad Pro 13 真机；尚未检查实际 safe area、DPR、Safari/PWA 字体、Pointer Events 和系统手势。
- A2759 目前也仍需在完整孩子可见流程完成后做一次最终真机验收。开发阶段的 1194×834 浏览器验收不能替代这一步。
- 竖屏目前只有方向提示，不能记录为“竖屏版已完成”。
- `scale < 0.72` 是当前保护阈值，未来支持 Split View 时需要根据真实触控尺寸重新验证，不能直接视为最终产品规则。

## 4. 完整版上线后恢复多尺寸适配的顺序

1. 冻结当时 A2759 横屏版本，保存所有孩子可见页面和关键状态的 1194×834黄金截图。
2. 明确下一批真实目标设备及 CSS 视口，不使用“所有 iPad”这种不可验收的范围。
3. 先恢复 iPad Pro 13 横屏：复用现有缩放公式，只检查背景延展、字体、触控、安全区和全部页面，不改舞台内部布局。
4. 为每个目标尺寸重跑首页三状态、全部学习卡、完成页、鸡蛋面板、孵化、拖拽、群聊、救援、减少动态和离线冷启动。
5. 浏览器自动截图通过后，再在对应真机的 standalone PWA 中检查 safe area、DPR、字体、触控与系统手势。
6. 每完成一个设备，把本文件的“当前结论”改为“已支持”，附新截图目录、测试日期、机型和系统版本。
7. 竖屏或窄窗口若需要真正可用，必须先形成独立视觉方案并由小皮确认；不得自动重排 F4 后直接宣称适配。
